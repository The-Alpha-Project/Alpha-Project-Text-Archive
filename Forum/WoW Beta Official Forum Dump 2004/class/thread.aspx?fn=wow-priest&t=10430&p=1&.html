I'm a software engineer, so what I'm going to tell you may only make sense in software engineer logic, but I'll tell you why what you're proposing likely doesn't work for the devs.

Basically what you're setting up is a complicated special case. Right now we have two cases, spells either stack or they don't. It's very simple, easy to understand, and there's not a lot of room for unforeseen side-effects (read: bugs).

Your rule would introduce third case that's complicated and hard to specify/test. This would be very error prone, potentially have many unforeseen side effects, and just generally turn a straightforward system into a morass of special cases that you'd need a law degree to understand.

Now, sometimes you do introduce special case logic into a program, but there has to be a really compelling reason to do it. It's a guilty until proven innocent thing, the burden is on the person who wants the special case to prove that it's vitally necessary to the system (game mechanics in this case).

I don't personally think you can make a case that two priests being able to stack renew in some very limited set of cases is vitally necessary to the game, but you're welcome to try.

I'm a software engineer as well.

This would be very easy to implement.

I'm not suggesting a specific case for priest renew only.

My suggestion is as follows:

All buffs overwrite each other, since, generally speaking, they are buffs, not 'effects' that last a short time.

Any short-time beneficial effect will stack with any others from different casters. However, there should be a hard limit as to how many beneficial (and negative) effects can be on a player at any one time.

So, if you had 4 priests and 3 druids all helping one tank, there could be 4 renews, 3 regrowths and 3 rejuvenates all at one time.

However, if the hard limit was 30, and you had 25 priests and 25 druids, then the limitations would kick in.

It just has to be designed and thought through. I know there is already code to limit how many debuffs/negative effects can be cast on an enemy. The same logic would apply here.

Design is the key.
